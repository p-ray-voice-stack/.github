# P-Ray Voice Stack

**LANGUAGE:** [English Version](./README.en.md)

> **P-Ray Voice Stack** 是一个面向企业级语音系统与交付工程的 GitHub 组织。  
> 我们优先解决的不是“做一个演示”，而是把语音相关能力组织成可交付、可部署、可维护、可持续演进的 to-b 系统。

## 定位

`p-ray-voice-stack` 不是单一产品仓库，也不是单纯的研究存档区。  
它的职责更接近一个**商业化语音系统工作区**：把开源主线、官网系统、镜像发布、部署契约、运维 handoff 和交付边界拆开管理，并让每一条线都能被后来者复核与接续。

我们特别强调：

- **to-b 系统的可信度，来自边界清晰，而不是功能堆叠**
- **基础设施、发布链路与宿主契约，本身就是产品能力的一部分**
- **“可部署”与“已部署”之间必须有证据链，而不是靠口头状态**

## 我们当前在做什么

当前组织内的工作重点主要集中在以下几类问题：

- **语音相关应用系统的架构与运行时设计**
- **官网前端、官网后端与公共访问入口的工程化建设**
- **容器镜像、宿主机、反向代理与发布路径的部署契约**
- **开源主线、官网运行线、商业交付线之间的仓库与责任分层**

如果一个系统未来要进入客户环境，或者要被团队反复交接，那么它就不能只停留在“能跑起来”。  
它必须回答这些问题：

- 仓库为什么这样分层
- 镜像为什么从这个路径发布
- 运行时为什么绑定这个宿主 contract
- 哪些部分属于公开主线，哪些部分属于私有交付或运维边界

## 仓库角色

当前组织下的仓库大致分为以下几类角色：

- **`P-Ray-Voice-Stack`**
  - 开源主线仓库
  - 承载公共系统工作、协议形状、运行时边界与长期主线实现

- **`pray-site-frontend`**
  - 官网前端仓库
  - 面向公开网站展示层、内容承载层与前端体验层

- **`pray-site-backend`**
  - 官网后端仓库
  - 面向网站 API、容器镜像、运行时宿主契约与后台部署线

- **其他交付或环境特定仓库**
  - 当某个运行时、客户环境或交付面不应与开源主线混放时，会保持独立仓库边界

这不是临时组织方式，而是有意识的商业化分层：  
**不是所有代码都应该进入同一仓库，也不是所有责任都应该共享同一条发布线。**

## 我们重视什么

在 to-b 语音系统里，我们更重视以下能力，而不是表面的“功能很多”：

- **明确契约**
  - 容器、端口、域名、代理、镜像 tag 与环境变量必须可解释

- **可复现发布**
  - 镜像路径、版本标识、compose 结构与宿主机 contract 必须稳定

- **可交接运行**
  - handoff 文档、部署边界、私有运维材料必须能支持后来者接手

- **公开与私有边界**
  - 开源主线、官网运行线与交付运行线不能长期混写成一团

- **狭窄检查点**
  - 我们偏好小而清楚的基础设施与运行时 checkpoint，而不是“一步到位”的黑箱式改动

## 协作方式

我们欢迎**谨慎、工程化、可解释**的协作方式。  
在这个组织里，好的贡献通常会改善这些事情：

- 仓库边界更清晰
- 发布链路更可靠
- 运行时更容易复核
- 官网与交付线更容易维护
- 文档与交接材料更接近真实状态

如果你正在阅读这个首页，最重要的一点不是“我们做了多少”，而是：  
**我们希望每一条系统线索都能在商业化推进过程中保持可解释、可交接、可安全演进。**
